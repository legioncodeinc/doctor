## Summary

PRD-004a and PRD-004b are substantively complete: all 8 a-AC and all 7 b-AC criteria are met with direct code and test evidence, the module-level non-goals are honored, and the served `/status.json` shape matches exactly what `hive`'s `fetchFleetStatus` consumer expects (field names and `health` enum values line up). Regression-checked against the real `main` baseline (493 tests / 49 files, verified in an isolated worktree), the branch adds tests without deleting or weakening any existing one. During this audit I found and fixed one Warning-level defect: every registered daemon's remediation ladder shared the same escalation hook, so a non-primary daemon's escalation (e.g. nectar giving up) could silently overwrite honeycomb's shared `needs-attention.json` dashboard read-seam, corrupting the escalation value b-AC-1/b-AC-2 require to be "that daemon's" own. This is now isolated (only the honeycomb entry writes the shared file; every other entry's escalation remains durably recoverable from its own `incidents-<name>.ndjson` shard) and covered by a new regression test. A second, lower-impact Warning (existing single-daemon installs lose `state.json`/`incidents.ndjson` continuity on upgrade to the sharded `state-honeycomb.json`/`incidents-honeycomb.ndjson` files) is documented as an accepted risk with a recommended follow-up, since a safe migration requires a dedicated design decision outside this audit's blast radius. No Critical findings. The gate is green after the fix (typecheck clean, 523/523 tests, build clean). **Recommend shipping.**

## Warnings (should fix)

- [x] **FIXED - Escalation cross-contamination: every daemon's ladder shared one `needs-attention.json` writer** - `src/compose/index.ts:467-500, 554-559` (pre-fix: `src/compose/index.ts:474-478, 536` in the original diff)

  `needsAttention` (`src/compose/index.ts:411-415`, unchanged) backs a single process-global `needs-attention.json` file - the pre-existing honeycomb dashboard read seam (`src/escalation/needs-attention-store.ts:7-14`), not sharded per daemon. Before this fix, the SAME `escalationHook` closure (old `src/compose/index.ts:474-478`) was wired into every entry's ladder (old `src/compose/index.ts:536`), so when a NON-primary daemon's ladder gave up (e.g. nectar exhausting its restart threshold), its escalation record would silently overwrite honeycomb's `needs-attention.json` - corrupting the "honeycomb" row's escalation value that both doctor's own status page (`readDaemonStatusRows`, `src/compose/index.ts:631-641`) and the real honeycomb daemon's dashboard read directly. b-AC-1 requires the per-daemon array to carry "that daemon's latest escalation," and b-AC-2 requires each entry's reported value to come from that daemon's own isolated shard, not a shared value cross-contaminated by a sibling daemon - this violated the spirit of both for the honeycomb row specifically. `EscalationRecord` (`src/rungs/escalation.ts:44-59`) carries no daemon-name field, so nothing downstream could distinguish whose escalation was being recorded.

  **Remediation applied:** split the shared hook into a per-entry factory (`buildEscalationHookFor`, `src/compose/index.ts:481-500`) that only calls `needsAttention.record()` for the `"honeycomb"` entry; every other entry still reaches the hosted telemetry sink (useful signal regardless of which daemon escalated) but never touches the shared file. Non-primary daemons' escalations remain fully recoverable via their own `incidents-<name>.ndjson` shard (the ladder's `heal()` in `src/supervisor.ts:205-219` already appends an "escalate" step to the per-entry incident builder, which `readPerDaemonEscalation` (`src/compose/index.ts:192-241`) scans back out for the status-page/CLI row) - no functional loss for the non-primary path, just isolation of the shared write.

  Also exposed `ladders: readonly RemediationLadder[]` on the `HiveDoctor` interface (`src/compose/index.ts:357-369, 724`), mirroring the existing `supervisors` field, so a test can drive a non-primary entry's `escalate()` directly.

  New regression test: `tests/compose/multi-daemon.test.ts` "escalation isolation: a non-primary daemon's escalate() never writes the shared needs-attention.json" - asserts a non-primary daemon's `ladder.escalate()` reaches the hosted sink but does not create `needs-attention.json`, and that the primary honeycomb entry's `escalate()` still writes it as before (backward-compatible).

  ```ts
  // src/compose/index.ts (after fix)
  const buildEscalationHookFor = (entryName: string): EscalationHook => {
  	return async (record): Promise<void> => {
  		if (entryName === "honeycomb") {
  			needsAttention.record(record);
  		}
  		await hostedEscalation(record);
  	};
  };
  // ...
  const entryLadder = createRemediationLadder({
  	rungs: [entryRestartRung, reinstallRung, uninstallRung],
  	restartGiveUpThreshold: entry.restartGiveUpThreshold,
  	logger,
  	escalationHook: buildEscalationHookFor(entry.name),
  });
  ```

- [ ] **ACCEPTED RISK - Existing single-daemon installs lose `state.json`/`incidents.ndjson` continuity on upgrade** - `src/state.ts:128-175` (`createStateStore`), `src/incidents.ts:139-209` (`createIncidentLog`), called with `name: entry.name` unconditionally at `src/compose/index.ts:538-539` and `src/cli/index.ts:108,113`

  Every production call site now passes a `name` (even the a-AC-2 single-daemon fallback resolves `name: "honeycomb"`, `src/compose/index.ts:132-142` `honeycombEntryFromConfig`), so the "omitted for the legacy single-daemon store" un-sharded path documented in `state.ts:66-72`/`incidents.ts:83-90` is never exercised in production - every install now reads/writes `state-honeycomb.json`/`incidents-honeycomb.ndjson`. An existing pre-PRD-004a installation upgrading to this version will find its historical `state.json` (`lastKnownHealth`, `consecutiveRestartFailures`, `backoffRung`, `lastHealAt`) and `incidents.ndjson` orphaned: the code now reads the new sharded paths, which do not exist yet, and defensively (and silently) falls back to `DEFAULT_STATE` / an empty incident tail rather than the box's real history.

  This does not violate a-AC-2's literal text ("a missing file must not wedge the watchdog") - the fallback is safe, not a crash, and `consecutiveRestartFailures` resetting to 0 fails toward leniency (a few extra free restarts before escalating again), not toward a worse crash-loop. It is, however, a real "additive, not disruptive" gap relative to a-AC-2's framing (see `src/compose/index.ts:132-142` `honeycombEntryFromConfig`, the a-AC-2 fallback entry that still resolves `name: "honeycomb"`), and no test in the suite asserts continuity across this exact migration boundary.

  **Why not fixed in this pass:** a correct migration needs a scoped design decision (a one-time rename/copy of `state.json` → `state-honeycomb.json` at boot, gated so it never fires for any other daemon name, and never overwrites an already-existing sharded file) that is more appropriately made deliberately (e.g. by `library-worker-bee` as a small follow-up PRD amendment or issue) than patched under audit-time pressure into a well-tested, security-reviewed defensive-read/write primitive. Recommend a small follow-up: on boot, if `entry.name === "honeycomb"` and `state-honeycomb.json`/`incidents-honeycomb.ndjson` do not yet exist but the legacy `state.json`/`incidents.ndjson` do, migrate them once (copy, don't delete the original) before constructing the entry's stores.

## Suggestions (consider improving)

- [ ] **Three near-duplicate "resolve daemons + honeycomb fallback" implementations** - `src/registry.ts:98-108, 245-250` (`honeycombFallbackEntry`, `loadRegistry`), `src/compose/index.ts:132-142, 251-259` (`honeycombEntryFromConfig`, `resolveDaemons`), `src/cli/index.ts:91-103` (inline fallback array)

  `registry.ts` exports `loadRegistry`/`honeycombFallbackEntry` as "the boot-time convenience" per its own docstring (`registry.ts:239-244`), but neither is called from any production path - `compose/index.ts` and `cli/index.ts` each independently re-implement "read the registry file, fall back to a honeycomb entry" using `config`-derived defaults (so env-var overrides on `healthUrl`/etc. are preserved), which is subtly different from `honeycombFallbackEntry`'s bare `DEFAULTS`-only fallback. `loadRegistry` is exercised only by `tests/registry.test.ts` in isolation. Not a bug today (every real call site is correct and tested), but three parallel implementations of the same fallback policy risk drifting apart silently as the registry schema evolves.

  Suggested: either wire `compose/index.ts`/`cli/index.ts` through a single parameterized `loadRegistry({ registryPath, home, fallback })`-style helper, or remove the unused `loadRegistry`/`honeycombFallbackEntry` exports and keep the config-aware fallback as the one canonical implementation.

- [ ] **`options.probe` test seam silently defeats per-entry probing when combined with multi-entry `daemons`** - `src/compose/index.ts:535-536`

  `entryProbe` falls back to `options.probe ?? (() => probeHealth({ healthUrl: entry.healthUrl, ... }))` - if a test (or future caller) supplies BOTH a multi-entry `daemons` array AND an `options.probe` override, every entry would silently share the exact same probe function regardless of its own `healthUrl`, defeating a-AC-1's "each probing its own healthUrl." Not currently reachable: single-daemon tests use `probe` overrides, and multi-daemon tests (`tests/compose/multi-daemon.test.ts`) rely on real per-entry mock servers without a `probe` override, so today's suite never exercises the combination. Suggested: scope `options.probe` to the primary entry only (matching how `options.readDaemonPid`/`options.restart` are already entry-parameterized) or document the incompatibility inline so a future test author does not accidentally write a false-positive isolation test.
