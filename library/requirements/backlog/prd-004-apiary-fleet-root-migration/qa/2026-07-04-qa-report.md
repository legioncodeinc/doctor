## Warnings (should fix)

- [ ] **PRD-004a legacy install-lock staleness check not implemented**, `src/apiary-migration.ts:36`, `src/install-lock.ts` (untouched)

  PRD-004a's proposed design (prd-004a, "Proposed design", line 55) states the install lock is deliberately not migrated as a live file AND that "a lock present in the legacy dir is honored by a legacy-fallback staleness check ... no code change beyond the fallback check". The exclusion half is implemented (`NOT_MIGRATED = new Set(["install.lock"])`, verified by the a-AC-4 test), but `src/install-lock.ts` is unchanged and no caller performs a legacy-dir lock check. During the window, a still-running pre-migration doctor process holding `~/.honeycomb/doctor/install.lock` is invisible to a new doctor acquiring at `<root>/doctor/install.lock`, so two remediation/reinstall flows could run concurrently. No AC covers this, so tests pass; it is a plan-vs-code gap, not an AC failure. Remediate by adding the legacy-fallback staleness check to lock acquisition (window-only, with a `LEGACY-HONEYCOMB-WINDOW` marker), or record an explicit decision that the risk is accepted.

  ```ts
  // src/apiary-migration.ts:36 (the exclusion exists; the companion fallback check does not)
  const NOT_MIGRATED = new Set(["install.lock"]);
  ```

- [ ] **Workspace migration ignores `DOCTOR_WORKSPACE_DIR`**, `src/apiary-migration.ts:102-104`, wired at `src/cli/index.ts:282`

  `migrateDoctorWorkspace` always computes source `<home>/.honeycomb/doctor` and destination `<root>/doctor` and never consults `env.DOCTOR_WORKSPACE_DIR`, while `resolveConfig` (a-AC-7, `src/config.ts:187`) honors the override for every read/write. Edge case: an operator who pinned `DOCTOR_WORKSPACE_DIR` to the legacy `~/.honeycomb/doctor` (to keep the old location) gets their live workspace files (state shards, incident logs) moved to an unused `~/.apiary/doctor/` on the first `doctor run` boot, and the configured workspace then reads empty. The state stores are defensive so nothing crashes, but supervision history is silently displaced. The PRD does not specify migration behavior under the override, so this is an alignment/robustness drift rather than an AC failure. Remediate by skipping the workspace migration when `DOCTOR_WORKSPACE_DIR` is set and non-blank.

  ```ts
  // src/apiary-migration.ts:102-104: env is used only for the fleet-root chain, never for the override
  const newDir = doctorStateDir(resolveApiaryRoot(env, home, platform));
  const legacyDir = join(legacyHoneycombRoot(home), "doctor");
  ```

- [ ] **c-AC-8 only partially proven: no Windows drive-letter re-anchoring test, host-native shapes only**, `tests/apiary-supervision.test.ts:86-145`, `tests/safe-path.test.ts:66-87`

  c-AC-8 requires containment "proven per root with `assertWithinBase` semantics on both POSIX and Windows path shapes", and the 004c design call-out (prd-004c line 61) explicitly names "on Windows, drive-letter re-anchoring" as a case to re-verify. The new suite proves traversal (`..`), relative paths, another product's subdir, and outside-every-root rejection, but all paths are built with the host's `path.join`, so any single run exercises exactly one path dialect, and no test presents a different-drive absolute path (for example `D:\evil\x.sqlite` against a `C:` root) to `coerceTelemetryDbPath`. The implementation looks correct by construction (`resolve` pins the drive at `src/registry.ts:273`; `assertWithinBase` prefix-checks with a trailing separator at `src/safe-path.ts:46-51`), so this is a test-coverage gap against the AC's own wording, not a suspected code defect. Remediate with an explicit drive-letter re-anchoring case (guarded to win32 hosts, or by exercising the rejection branch with a synthetic absolute path that resolves off-root on both dialects).

## Suggestions (consider improving)

- [ ] **Stale doc comments still present the legacy locations as current**, `src/compose/index.ts:320`, also `src/compose/index.ts:272,456,680`, `src/device-id.ts:7-13,143`, `src/telemetry/capture.ts:34`

  The `CreateDoctorOptions.registryPath` doc says "default: `~/.honeycomb/doctor.daemons.json`" but the default is now the two-location fleet resolution rooted at `<root>/registry.json`; the `device-id.ts` and `capture.ts` module headers likewise still narrate `~/.honeycomb/device.json` / `~/.honeycomb/install-id` as the primary locations. Function-level docs are correct; only these older narrations drifted. Misleading for the next editor.

- [ ] **Dead exports**, `src/registry.ts:545` (`loadFleetRegistry`), `src/device-id.ts:47` (`honeycombHomeDir`)

  `loadFleetRegistry` (migrate-then-resolve convenience) has no callers and no tests; the boot path calls `runApiaryMigrations()` then `createDoctor` -> `resolveRegistryEntries` separately (`src/cli/index.ts:282`). `honeycombHomeDir` no longer has any consumer (`capture.ts` imports `legacyHoneycombRoot` directly). Either wire `loadFleetRegistry` in or drop both.

- [ ] **Needs-attention diagnosis names the new default path even when the legacy file is the malformed one**, `src/compose/index.ts:490-494`

  When `resolveRegistryEntries` throws on a malformed legacy file, the surfaced record says "registry at `<root>/registry.json` ... malformed". Impact is low because the embedded `RegistryError` message carries the actual offending path (`src/registry.ts:347-366`), but the header path can misdirect an operator mid-window.
