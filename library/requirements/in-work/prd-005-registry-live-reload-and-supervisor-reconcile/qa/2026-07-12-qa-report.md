## Suggestions (consider improving)

### S-1 — Reload ignores the boot-time `options.registryPath` / `options.daemons` seams (acceptable)

**File:** `src/compose/index.ts:860-864` (reload loop construction) vs `src/compose/index.ts:295-325` (`resolveDaemons` boot seams)

At boot, `resolveDaemons` honors two injectable test seams: `options.daemons` (a direct array) and `options.registryPath` (a single explicit file). The reload loop is constructed with only `{ home, env, platform }`, so its default `resolveEntries` calls `resolveRegistryEntries({ home, env, platform })` — the two-location resolution — and does **not** consult those seams. A hypothetical test that injects `options.registryPath` at boot would see boot read the custom path while reload reads `<root>/registry.json`.

**Verdict: acceptable, not a defect.** (a) 005a **a-AC-3 literally specifies** the call `resolveRegistryEntries({ home, env, platform })`, so the implementation is AC-conformant by mandate. (b) Production **never** injects `daemons`/`registryPath` (those seams exist only for hermetic tests / operator override), so boot and reload read the identical file in every real deployment — there is no production divergence. (c) The compose e2e test (`tests/compose/registry-reload-reconcile.test.ts:104-107`) deliberately avoids the seams and writes the real `<home>/.apiary/registry.json` so boot and reload share one file. No AC and no production path is affected. Left as a Suggestion only so a future maintainer who adds a seam-based test is aware reload won't track it.

### S-2 — No integration test exercises the `armDaemon` running-branch for a runtime-added daemon

**File:** `src/compose/index.ts:698-707` (`armDaemon`); `tests/compose/registry-reload-reconcile.test.ts`

`armDaemon` starts a reconcile-added supervisor's watch loop via `built.supervisor.start()` only when `running` is true; when doctor is not yet running it defers (boot's `start()` arms the whole `builtMap`). The compose e2e tests never call `doctor.start()` — they boot, drive `registryReloadLoop.tick()`, and assert set membership / status rows / telemetry — so the `running === true` branch of `armDaemon` (the actual `supervisor.start()` for a daemon added *after* boot) is not directly asserted at the integration level.

**Verdict: Suggestion only.** The behavior is covered where the plan required it: the 005b reconcile unit test proves reconcile calls `armDaemon` (with startup grace armed first, `tests/registry-reconcile.test.ts:128-136`), and the compose→`start()` wiring for the initial set is exercised by pre-existing compose tests. The `armDaemon` adapter is a thin, correct three-line push onto `supervisorRuns` mirroring boot. b-AC-1's PRD-specified test exists and passes, so this is not an AC gap — only a nice-to-have hardening (start doctor, tick a reload that adds a daemon, assert its loop spins).
